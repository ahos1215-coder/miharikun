# 国交省スクレイピング再設計計画 — Gemini レビュー依頼

> **作成者**: Claude Opus 4.6（副船長 / シニアアーキテクト）
> **作成日**: 2026-04-01
> **目的**: 国交省からの情報取得を「航海士が本当に必要な情報だけを蒸留する」設計に刷新する

## 1. 現状の問題（正直な自己批判）

### 1-1. DBに入っている国交省データの実態

現在 regulations テーブルにある MLIT ソースのデータの**大半が知床遊覧船事故関連の旅客船安全対策**です。外航船・内航船の航海士にとって**ほぼ無関係なノイズ**が DB を埋めています。

```
実例（全て Critical 判定で DB に保存済み）:
- 「旅客船の安全・安心対策」
- 「海事安全規制強化と制度改革の方向性」
- 「事故対策検討報告書」
- 「旅客船の総合的な安全・安心対策」
- 「海事安全対策の方向性」
→ 全て知床遊覧船事故の検討委員会資料。外航バルカーの航海士には無関係。
```

### 1-2. なぜこうなったか

| 経路 | 問題 |
|------|------|
| **RSS** (`pressrelease.rdf`) | 国交省全体のプレスリリースを取得。27キーワードでフィルタしているが、「船舶」「安全」等の広いキーワードで旅客船事故関連も通過してしまう |
| **BFS クローラー** | 409バグで**一度も完走していなかった**（修正済みだが再実行前）。つまり施策ページからの情報は DB に一切入っていない |

### 1-3. 本当に必要だった情報

航海士が必要としているのは以下のような「施策ページ（ストック情報）」:

- `maritime_fr4_000055.html` — **基本訓練（令和8年2月14日適用）**: 船員法改正に伴う新制度。STCW 免状維持に直結
- `maritime_fr4_000030.html` — **船員安全衛生**: 閉囲区画立入、フルハーネス等の現場実務
- `maritime_fr4_000017.html` — **海上労働条約（MLC）の批准に伴う船員法改正**
- `maritime_fr7_000019.html` — **SOx 規制への対応**: MARPOL Annex VI、燃料油規制
- `maritime_fr8_000061.html` — **救命いかだ搭載義務化**: 設備改修に直結

**これらは RSS には流れない。施策ページとして静的に存在し、内容が更新される。**

## 2. 再設計案: 3層アーキテクチャ

### 第1層: RSS フィルタの厳格化（ノイズ排除）

```
現状: 27キーワードで包含フィルタ → 旅客船事故も通過
改善: 包含フィルタ + 除外フィルタの2段階

除外キーワード（新設）:
  旅客船, 遊覧船, 知床, 観光船, 観光バス, フェリー事故,
  造船, 舶用工業, 産業振興, 造船業,
  入札, 予算, 人事, 統計, 決算, 組織改編,
  港湾施設, 岸壁, 埠頭, 防波堤, ターミナル,
  検討会開催, 議事録, 委員会議事, パブリックコメント募集結果

包含キーワード（既存27個に追加）:
  基本訓練, 船員法改正, STCW改正, 海技免状,
  閉囲区画, enclosed space, 安全管理規程
```

**効果**: 旅客船事故関連は除外しつつ、船員法・STCW・設備関連のプレスリリースは引き続き捕捉。

### 第2層: シードURL方式の施策ページ監視（新設）

BFS クロールを廃止し、**重要なインデックスページを直接巡回 → リンク先の施策ページを深さ1で探索**する。

```
シードURL（インデックスページ、6件固定）:
  1. /maritime/maritime_mn4_000005.html  — 運航労務監理（13施策リンク）
  2. /maritime/maritime_tk8_000003.html  — 船舶の安全・環境（14施策リンク）
  3. /maritime/maritime_fr4_000030.html  — 船員安全衛生（12施策リンク）
  4. /maritime/maritime_tk4_000016.html  — 船員の現状
  5. /maritime/maritime_tk10_000017.html — 船員養成
  6. /maritime/maritime_fr1_000027.html  — 法律（所管法令PDF）

巡回フロー:
  Step 1: シードURL 6ページを取得（18秒）
  Step 2: 各ページから maritime_fr* / maritime_tk* のリンクを全抽出（推定40-60件）
  Step 3: 各施策ページを訪問し、<main> 要素のハッシュを計算
  Step 4: mlit_crawl_state テーブルの前回ハッシュと比較
  Step 5: 差分あり → 新規PDFリンクを抽出 → pending_queue に登録
  Step 6: 差分なし → スキップ

ノイズ除外（URLパターン + タイトルキーワード）:
  URL除外: /budget/, /statistics/, /bidding/, /event/, /personnel/
  タイトル除外: 入札, 統計, 予算, 人事, イベント, 説明会開催案内
```

### 第3層: Discovery Mode（未知の施策ページ自動発見）

```
Step 2 で抽出した maritime_fr* URL のうち、
mlit_crawl_state に存在しないものを「新規ポータル発見」として記録。

週次で Gemini に鑑定させる:
  「このページは航海士の業務に影響する重要施策か？」
  → Yes: 監視対象に昇格
  → No: 除外リストに追加

Gemini API 消費: 新規ポータル数件/週 × 1回 = 極小
```

## 3. `<main>` 要素ハッシュ監視の詳細

### なぜ HTML 全体ではなく `<main>` だけか

国交省のサイトは共通ヘッダー・フッター・サイドメニューが頻繁に微修正される（バナー追加、リンク順序変更等）。HTML 全体のハッシュだと、**内容が変わっていないのに「更新あり」と誤検知**する。

```python
# 現状（全体ハッシュ）
page_hash = hashlib.sha256(html_content.encode()).hexdigest()
# → ヘッダーのバナー変更でも差分検知 → 誤検知

# 改善案（<main> 要素のみ）
soup = BeautifulSoup(html_content, "html.parser")
main = soup.find("main") or soup.find("div", id="content") or soup.find("div", class_="content")
if main:
    # スクリプトタグやスタイルタグを除去
    for tag in main.find_all(["script", "style", "noscript"]):
        tag.decompose()
    content_hash = hashlib.sha256(main.get_text(strip=True).encode()).hexdigest()
```

### 差分が見つかった時の処理

`difflib` は使わない。変更があった事実だけで十分。

```
差分あり → ページ内の PDF リンクを全抽出
  → 新規 PDF は pending_queue に登録（Gemini 分類は process-queue に委任）
  → 既知 PDF はスキップ
  → mlit_crawl_state のハッシュを更新
```

## Gemini への質問事項

### Q1: シードURL方式 vs BFS の判断

シードURLを固定リストで管理する方式に切り替えますが、**国交省がサイト構造を変更した場合（URL変更、ページ統廃合）にどう対応すべきか？** 半年に1回の手動チェックで十分か、それとも自動検知の仕組みが必要か？

### Q2: RSS の除外キーワード設計

「旅客船」を除外すると、**旅客船を運航している会社の航海士には不利益**になる可能性がある。ユーザーの船種（ship_type: passenger）に応じて除外キーワードを動的に切り替えるべきか？それとも「旅客船安全対策の検討委員会資料」に限定して除外すべきか？

### Q3: `<main>` 要素が存在しないページへの対応

国交省の古いページには `<main>` タグがない場合がある。フォールバックとして `div#content` や `div.content` を使う予定だが、これで十分か？他に推奨されるセレクタはあるか？

### Q4: Discovery Mode の鑑定プロンプト設計

新規発見した施策ページを Gemini に鑑定させる際、**「航海士にとって有益か？」の判断基準**をどう構造化すべきか？以下の案で問題ないか？

```
鑑定基準（案）:
- 船員法・海技免状に直接影響する制度変更 → 有益
- 船舶設備（救命/消防/航海計器）の基準変更 → 有益
- ISM/ISPS の運用変更 → 有益
- MARPOL 排出規制の変更 → 有益
- 港湾施設・造船所向けの基準 → 除外
- 行政組織・人事・予算の変更 → 除外
- 旅客船に限定された規制（外航船に影響なし） → 条件付き除外
```

### Q5: 関東運輸局の有用性評価

3ヶ月後に有用性を評価する予定だが、**評価指標として何を見るべきか？** 以下で十分か？

- 本庁にない新規情報の件数
- ユーザーのフィードバック（👍率）
- ノイズ率（有益/全体の比率）

### Q6: 既存ノイズデータの扱い

DB に入っている旅客船事故関連データ（推定20-30件）を一括削除するか、非表示化するか。**法的リスク（情報の欠落）と UX（ノイズ排除）のトレードオフ**についてアドバイスが欲しい。

### Q7: 国交省以外で見落としているソース

NK + 国交省 + e-Gov の3ソースでカバーしているが、**日本の航海士にとって本当に重要な情報源で見落としているもの**はないか？以下は検討済みだが不採用としたもの:

- IMO 直接取得 → ClassNK 経由で十分と判断
- 海上保安庁 → 水路通報は書籍監視でカバー
- 日本海事協会 (JG) → NK がカバー

他に「これは拾うべきだ」というソースがあれば教えてほしい。
