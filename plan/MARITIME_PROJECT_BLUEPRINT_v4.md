## 1. プロダクト概要

### 1.1 一言で

**自船のスペックと乗組員の資格情報を登録するだけで、NK・IMO・国交省の規制更新の中から「自分の船と乗組員に関係あるもの」だけを自動抽出して届けるサービス。**

### 1.2 対象とする規則の領域

| 領域 | 主な条約・法令 | 例 |
|------|-------------|-----|
| **船舶安全** | SOLAS, 船舶安全法 | 浸水警報装置の設置義務拡大, 揚貨装置の新要件 |
| **海洋環境** | MARPOL, CII/EEXI, EU ETS, UK-ETS | DCS 報告項目追加, PFOS 消火剤禁止 |
| **船員資格** | STCW, STCW-F, 船員法 | ハラスメント防止訓練義務化, 電子証明書, 内航船拡大 |
| **安全管理** | ISM Code | 安全管理システム規則の改正 |
| **保安** | ISPS Code | 保安要件の変更 |
| **船級規則** | NK Rules, IACS UR | ClassNK テクニカルインフォメーション |
| **船員労働** | MLC, 船員法 | 労働条件の改正 |
| **検査・PSC** | Tokyo MOU, JG検査, 各旗国通達 | 重点検査キャンペーン, 検査心得改正 |

### 1.3 ターゲットユーザー

- **第一ターゲット**: 航海士・機関士（自分の資格 + 船の規則を知りたい）
- **第二ターゲット**: 船舶管理会社の工務・海務担当者
- **将来**: 海外船籍、タンカー・客船、漁船（STCW-F）

## 3. 監視ソース

### 3.1 MVP — 主要ソース

| ソース | 対象領域 | 取得方法 |
|--------|---------|---------|
| **ClassNK テクニカルインフォメーション** | 全領域 | HTML一覧 + PDF → Gemini |

| **IMO ニュース / 公開 Circular** | SOLAS, MARPOL, STCW | ニュースフィード + PDF |

### 3.2 Phase 2 追加

海上保安庁, 東京MOU(PSC), Japan P&I Club, 地方運輸局, JMETS

### 3.3 国交省 3層スクレイピング戦略

#### 第1層: 公式RSSフィード（確実・即時・日次）

国交省のプレスリリースRSS・新着情報RSSを監視。キーワードフィルタで海事系を抽出し、リンク先ページと添付PDFをGeminiに送信して分類。

#### 第2層: `/maritime/` 全体クロール（網羅性・週次）

`mlit.go.jp/maritime/` 配下のリンクを再帰的に辿り、前回との diff で新規 URL・PDF を検出。URLパターンに依存しないため、サイト構造変更に強い。

**PDF 品質プリプロセス（v4 追加）**: 検出した PDF に対して以下を事前チェック:
- `Content-Length` / `Last-Modified` ヘッダーで変更の有無を判定（未変更ならスキップ）
- ファイルサイズが異常に小さい（< 1KB）→ リンク切れの可能性 → スキップ + アラート
- PyMuPDF でテキスト抽出を試行 → テキストが極端に少ない場合は **スキャン画像PDF** と判定 → OCR 処理（Gemini はある程度の画像PDFも読めるが、品質が低い場合は精度低下の警告を付与）

#### 第3層: e-Gov パブコメ監視（先行把握・週次）

省令改正案を施行前に捕捉。`severity: "upcoming"` で分類。

#### JG検査の対応

船舶安全法施行規則の改正 → 第1層で捕捉。船舶設備規程の改正 → 第1層 + 第2層。検査心得の改正 → 第2層。省令改正案 → 第3層で先行把握。

## 5. AI 分類の信頼性設計（v4 新設）

### 5.1 Confidence Score

Gemini の分類プロンプトに `confidence` フィールド（0.0〜1.0）を追加。

| スコア | ラベル | UI 表示 | 運用 |
|--------|--------|---------|------|
| 0.8〜1.0 | 高確度 | 通常表示 | 自動公開 |
| 0.5〜0.79 | 中確度 | 「⚠ 要確認」バッジ | 自動公開 + 注意喚起 |
| 0.0〜0.49 | 低確度 | 「❓ AI判定不確実」 | 管理者手動レビューキューへ |

### 5.2 原文引用（Citations）

AI が分類の根拠とした PDF 内の該当箇所を必ず返す。

```json
{
  "confidence": 0.85,
  "citations": [
    {
      "text": "本規則は、総トン数500トン以上の貨物船に適用する",
      "page": 2,
      "source": "TEC-1361"
    }
  ]
}
```

UI 上で分類結果の横に「根拠: TEC-1361 p.2」のリンクを表示。ユーザーがワンクリックで原文PDFの該当箇所を確認できるようにする。

### 5.3 ハルシネーション対策

- プロンプトに「過去の知識を使わず、添付PDFのテキストのみから判断せよ」を明記
- `temperature: 0.1` で創造性を抑制
- 月次で10件程度のサンプルを人間（開発者）が目視チェックし、精度を継続監視

### 5.4 免責事項

UI の全ページフッターおよび分類結果の表示時に以下を明記:

> **本サービスはAIによる参考情報の提供を目的としており、公式文書の代替ではありません。法令遵守の最終判断は、必ず原文を確認の上ご自身の責任で行ってください。**

利用規約にも同等の免責条項を記載。

## 7. DB スキーマ

### 7.1 regulations

```sql
CREATE TABLE regulations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source TEXT NOT NULL,               -- 'nk' | 'imo' | 'mlit'
  source_id TEXT NOT NULL,
  title TEXT NOT NULL,
  title_en TEXT,
  summary_ja TEXT,
  url TEXT,
  pdf_url TEXT,
  published_at TIMESTAMPTZ,
  domain TEXT DEFAULT 'ship',         -- 'ship' | 'crew' | 'management' | 'environment' | 'psc'
  applicable_ship_types TEXT[],
  applicable_gt_min INT,
  applicable_gt_max INT,
  applicable_built_after INT,
  applicable_routes TEXT[],
  applicable_flags TEXT[],
  applicable_crew_roles TEXT[],       -- v4: 船員職種の適用範囲
  category TEXT,
  severity TEXT DEFAULT 'informational',
  confidence FLOAT,                   -- v4: AI 分類の確度（0.0〜1.0）
  citations JSONB,                    -- v4: AI が根拠とした原文引用
  needs_review BOOLEAN DEFAULT false, -- v4: 低確度 → 手動レビュー待ち
  gdrive_text_file_id TEXT,
  scraped_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(source, source_id)
);
```

## 10. UX 設計原則（v4 新設）

### 10.1 船上の通信環境への配慮

船上では衛星通信（Inmarsat/VSAT）が主で、帯域制限・従量課金が一般的。

- **通知はテキストベース優先**: メール本文またはLINEメッセージに要約を完結させる。ダッシュボードを開かなくても必要な情報が得られる
- **ダッシュボードは軽量**: 画像・チャートを最小限に。テキスト中心のUI
- **PWA オフラインキャッシュ**: 最新の分類結果をローカルに保持（Phase 3）

### 10.2 AI 判定の透明性

- 分類結果の横に必ず「根拠」リンクを表示
- 低確度の分類には「⚠ AI判定が不確実です。原文を確認してください」を表示
- 免責事項を全ページフッターに配置

## 11. エラーハンドリング戦略（v4 新設）

### 11.1 明示的なエラー

| エラー種別 | 検知方法 | 対応 |
|-----------|---------|------|
| NK サイト構造変更 | パース結果が 0 件 | LINE 通知 + 前回データ維持 |
| 国交省 RSS 取得失敗 | HTTP 4xx/5xx | 3回リトライ → 失敗通知 |
| Gemini API エラー | 429/500 | 指数バックオフ → 未処理キューへ |
| Gemini 分類結果が不正 JSON | パースエラー | 再試行 1 回 → 手動キューへ |
| Supabase 書き込み失敗 | HTTP エラー | ローカル JSON 退避 → 次回再試行 |
| 国交省クロールで大量新規 URL（100+） | 閾値チェック | サイトリニューアル疑い → 通知して手動確認 |

### 11.2 サイレント障害（v4 追加 — Gemini レビューで指摘）

| シナリオ | 検知方法 | 対応 |
|---------|---------|------|
| **データの鮮度落ち**: ソースが長期間更新なし | 各ソースの最終更新日を監視。30日間新規 0 件 → アラート | 管理者がソース URL を手動確認 |
| **Gemini の先入観**: 新しい特例を過去の知識で見落とす | 月次 10 件サンプルの人間チェック | プロンプト修正 + 再分類 |
| **PDF リンク切れ**: 国交省が PDF を移動/削除 | ダウンロード時の 404 検知 | 対象規制に「原文アクセス不可」ラベル |

## 12. 自律運用設計（v4 追記 — 船乗りオペレーター前提）

> **設計原則**: 運用者（開発者）が数ヶ月間乗船中でも、サービスが自力で動き続ける。
> 壊れたら「通知して人間を待つ」ではなく「自動回復を試み、ダメなら安全に縮退する」。

### 12.1 自動回復（Self-healing）

| 障害 | 現v4の対応 | 改善: 自動回復 |
|------|-----------|---------------|
| NK サイト構造変更 | LINE通知 → 人間がコード修正 | ① 前回成功時のパース方式でリトライ → ② URL直接パターン（`T{番号}j.pdf`）でPDF取得を試行 → ③ 両方失敗なら「NK新着停止中」ラベルをUIに自動表示 → ④ 毎日リトライし続ける |
| Gemini API 429/500 | 指数バックオフ → 未処理キュー | ① 指数バックオフ（最大6回） → ② 未処理PDFを `pending_queue` テーブルに保存 → ③ 翌日のGHAバッチで自動再処理 → ④ 3日連続全失敗で初めて通知 |
| Gemini モデル廃止 | 人間がモデル名を変更 | ① 環境変数 `GEMINI_MODEL` で制御 → ② GHAワークフロー内でモデルの有効性チェック → ③ 失敗時はフォールバックモデル（`GEMINI_FALLBACK_MODEL`）に自動切り替え |
| 国交省 RSS 取得失敗 | 3回リトライ → 通知 | ① 3回リトライ → ② 失敗してもクロール（第2層）が独立して動くので片方が生きていれば捕捉継続 → ③ 両方7日連続失敗で通知 |
| Supabase 書き込み失敗 | ローカル JSON 退避 | ① JSON退避 → ② 次回バッチで自動再投入 → ③ 退避ファイルが10件超で通知 |

### 12.2 縮退運転（Graceful degradation）

**原則: スクレイパーが壊れても、既存データと資格管理は正常に動き続ける。**

```
正常運転                    縮退運転（スクレイパー障害時）
┌──────────────┐           ┌──────────────┐
│ 新着規制アラート│ → 停止    │ 「最終更新: 4/15」│ ← 鮮度を明示
│ AI 分類      │ → 停止    │ 既存分類データ   │ ← そのまま表示
│ 資格リマインダー│ → 継続    │ 資格リマインダー  │ ← 正常動作
│ ニュース閲覧  │ → 継続    │ ニュース閲覧    │ ← 過去分は閲覧可
│ ダッシュボード │ → 継続    │ ダッシュボード   │ ← 最終データで表示
└──────────────┘           └──────────────┘
```

UI への反映:
- 各ソースの「最終取得成功日時」をフッターに表示
- 7日以上更新がないソースは「⚠ NK: 7日間更新なし」のバナー表示
- 30日以上更新なしで「このソースは現在取得できていません。原文サイトを直接確認してください」+ 原文リンク

### 12.3 ヘルスチェック・ワークフロー（GHA）

通常のスクレイピングGHAとは**別に**、独立したヘルスチェック用ワークフローを毎日実行する。

```yaml
# .github/workflows/health-check.yml
# 毎日 JST 12:00 に実行。スクレイピング GHA の成否を外側から監視する。
name: Health Check
on:
  schedule:
    - cron: "0 3 * * *"  # UTC 03:00 = JST 12:00
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - name: Check last successful scrape
        run: |
          # Supabase から各ソースの最終取得日時を取得
          # 3日以上前 → LINE通知
          # 7日以上前 → LINE通知（緊急）
          # Gemini pending_queue の件数チェック
          # Supabase DB容量チェック
```

### 12.4 乗船前チェックリスト

乗船前に `/admin/health` ページで5分で確認:

| チェック項目 | 自動判定 | 表示 |
|------------|---------|------|
| GHA 過去7日の成功率 | ≥90% → OK | 緑/黄/赤 |
| 各ソースの最終取得日 | 3日以内 → OK | 緑/黄/赤 |
| Supabase DB 容量 | <400MB → OK | 使用量バー |
| Gemini 未処理キュー | 0件 → OK | 件数表示 |
| Gemini API 直近エラー率 | <5% → OK | 緑/黄/赤 |
| 証明書期限（自分） | 90日以上 → OK | 日数表示 |
| 直近のエラー通知 | 未解決0件 → OK | 件数 |

**全項目グリーンなら安心して乗船できる。**

### 12.5 通知の多重化

船上では LINE が届かない期間がある。通知を多重化して確実性を高める。

| 通知手段 | 用途 | 船上での到達性 |
|---------|------|-------------|
| **メール** | 重要アラート + 週次サマリー | VSAT 環境で受信可能（最も確実） |
| **LINE** | リアルタイム通知 | VSAT/港寄港時に受信 |
| **UI バナー** | 縮退運転の状態表示 | 次回アクセス時に確認 |

**週次サマリーメール**（自動生成、GHA）:
- 今週の新着規制件数（ソース別）
- 自分に関係ある規制の一覧（要約付き）
- システムヘルス（全ソースの取得状況）
- 次の証明書更新期限

→ これにより、船上でダッシュボードを開かなくても**メール1通で状況がわかる**。

### 12.6 想定される「人手が必要な介入」とその頻度

完全なメンテナンスフリーは現実的に不可能だが、介入頻度を最小化する。

| 介入 | 想定頻度 | 緊急度 | 乗船中の対応 |
|------|---------|--------|------------|
| NKサイト大規模リニューアル | 年1回程度 | 低（自動フォールバック中） | 下船後に対応 |
| Gemini モデル廃止 | 年1-2回 | 中（フォールバックモデルで継続） | 環境変数1つ変更（スマホから可） |
| Supabase/Vercel 無料枠変更 | 年0-1回 | 低 | 下船後に対応 |
| AI 分類精度の改善 | 四半期に1回 | 低 | 下船後にプロンプト調整 |
| 新しい監視ソースの追加 | 随時（要望次第） | 低 | 下船後に実装 |

**結論**: 自動回復 + 縮退運転 + ヘルスチェックにより、**通常は3-6ヶ月ノータッチで運用可能**。壊れた場合も「止まる」のではなく「古いデータで動き続けながら自動リトライする」設計。緊急の介入が必要なケースは年に数回程度で、その多くはスマホから環境変数を1つ変更するだけで対応できる。

## 14. 競合との戦い方

**ポジショニング**: 「OneOcean の廉価版」ではなく「日本の航海士の資格管理 + 規制アラートのパーソナルアシスタント」。

**勝てるポイント**:
1. **資格管理が入口**: 免状更新を忘れない安心感 → そこから規制アラートへ
2. **日本語 × 国交省の深さ**: OneOcean がカバーしきれない領域
3. **AI パーソナライズ**: 「自分の資格 × 自分の船」の掛け合わせ
4. **価格**: 個人が月額数千円で使える
5. **スピード**: 通達公開から数時間で分類完了

**当面は諦めるポイント**:
1. 50 旗国の網羅性 → 日本籍船特化
2. PSC 検査での公式信頼性 → 情報提供ツールとしての立場
3. 完全なオフライン → テキスト通知で代替
