## A. プロジェクトの全体像と方向性の検証

### 現在地
MIHARIKUN は「海事規制の自動収集・AI分類・パーソナライズ通知」サービスである。開発開始から 3 日で Phase 3 まで到達し、本番稼働中（https://miharikun2.vercel.app）。

### 規模
- 規制データ: 453件（ClassNK + 国交省 RSS/クロール + e-Gov パブコメ）
- DB: Supabase PostgreSQL, 8テーブル, 12マイグレーション適用済み
- GHA ワークフロー: 15本（スクレイパー5 + マッチング/分類3 + 書籍2 + CI/CD3 + 通知2）
- テスト: 121件（Python 112 + E2E 9）
- フロントエンド: Next.js 16, 18ルート, Glassmorphism ダークテーマ
- バックエンド: Python 3.11, 67書籍マスターデータ, 43条約ルール, 871キーワード

### ビジョン
「膨大な規制情報の中から、**自船にのみ関係ある情報を自動抽出・通知し、具体的な対応事項（船側/会社側）まで提示する**」

ターゲット: 個人航海士〜中小船舶管理会社（OneOcean 等の年間数十万円のサービスの代替）

### Gemini への問い
1. Phase 3 の現在の進捗（書籍管理・マッチングエンジン刷新・UI リデザイン）は、「日本の航海士を救う」というビジョンに最短距離で向かっているか？遠回りしている機能はないか？
2. Master Matching 方式（Gemini を「判定員」から「ルール作成者」に転換し、マッチングは DB ルール vs 船舶スペックの純粋比較）への移行により、ビジネスモデルにどのような変化が起きるか？例えば Free ティアの制限をどう設計すべきか？

## C. ハイエンド UI/UX の冷徹な批判

### 現在のデザイン方針
- v5 の「超軽量 UI（50KB・画像なし）」を v6 で明示的にオーバーライド
- Glassmorphism + ダークネイビー (#0a1628) + シアンアクセント (#06b6d4)
- Framer Motion アニメーション（staggered entrance、gauge animation）
- Lucide React アイコン（テキスト絵文字を全廃）

### 依存パッケージの重量
- `framer-motion`: 5.4MB（gzip 約200KB）— 10個のDOM要素アニメーション用
- `lucide-react`: アイコンセット

### 懸念
- 船上での実務：荒天、疲労、片手（スマホ）操作、夜間当直
- v5 で「初期ロード 50KB」を目指していたが、現在は制約を撤廃
- Glassmorphism の backdrop-blur は低スペック端末で重い可能性

### Gemini への問い
1. 「256kbps の制約を捨てた」判断は正しかったか？船上のVSAT環境では、リッチUIが逆にUXを損なう可能性はないか？
2. 「Maritime Command Center」のような重厚なダッシュボードは、実際の航海士が「当直の合間に5秒で確認したい」ニーズに合っているか？
3. **情報の過多**：43条約×67書籍×複数船のデータを1画面に詰め込んでいないか？ユーザーが迷子になるリスクは？

## D. 戦略的ピボットのアドバイス

### 現在の立ち位置
- 個人開発（1人 + AI）で 3 日で Phase 3 まで到達
- ユーザー: 開発者自身（航海士）+ 少数のテスター
- 収益: ゼロ（フリーミアムモデル未実装）
- 競合: OneOcean（年間 $1,000-5,000/隻）、Regs4ships

### Master Matching 方式への転換の意味
- API コスト: 船舶数に依存しなくなった → Free ティアの制限設計が変わる
- スケール: 100隻でも1000隻でも同じコスト → Fleet 管理会社向けの価格設定が容易に
- 技術的優位: OneOcean は人手キュレーション、MIHARIKUN は AI 自動分類+ルール自動抽出

### Gemini への問い
1. 「個人開発の延長」から SaaS 商用化へ移行するために、今すぐ変えるべき設計思想は？
2. Free/Pro/Fleet のティア設計: Master Matching 方式では API コストがスケールしないので、何をもって Pro/Fleet の差別化をすべきか？
3. 日本市場に閉じるべきか、英語対応して国際展開すべきか？タイミングは？
4. **船舶管理会社への営業**を想定した時、現在のプロダクトに最も不足しているのは何か？

## E. 技術的負債と効率性の監査

### 自己申告の技術的負債（P0-P2）

**P0（致命的）:**
1. Gemini 設定が 3 箇所で重複（MAX_RETRIES が 4 と 6 で不一致）
2. 日次 RSS スクレイパーに単体テストがない（578行、本番で毎日稼働）
3. フロントエンド単体テストがゼロ（30+コンポーネント）

**P1（高）:**
4. 船種定数が 2 ファイルで重複
5. extract/generate スクリプトが gemini_client を使わず独自 API 実装
6. publication_requirements.py が 2,258行で巨大
7. Python(67件) と TypeScript(43件) の書籍データ乖離

**P2（中）:**
8. CoVe（Chain of Verification）のコスト対効果が不明
9. framer-motion 5.4MB（10要素のアニメーション用）
10-13. デッドコード、magic numbers、URL ハードコード、ドキュメント陳腐化

### API Frugality の現状
- **Master Matching 導入済み**: マッチングは applicability_rules JSON で判定（API不要）
- **残っている API 消費ポイント**:
  - スクレイパーでの PDF 分類（取り込み時、必須）
  - extract_applicability_rules.py（初回 453件、以後は新規のみ）
  - generate_headlines.py（見出し生成、任意）
  - CoVe 検証（低確信度の再検証、任意）
  - process_queue.py（失敗リトライ、必須）

### アーキテクチャ図（現在）
```
[ユーザー] → [Vercel (Next.js 16)] → [Supabase (Auth + DB + RLS)]
                                            ↑
[GitHub Actions (15 workflows)]
  ├── スクレイパー: NK(日次) / MLIT RSS(日次) / MLIT Crawl(週次) / e-Gov(日次)
  ├── Gemini: PDF分類(取り込み時) / applicability_rules抽出(初回のみ) / headline生成
  ├── マッチング: applicability_rules vs 船舶スペック (API不要)
  ├── 書籍: seed(初回) / 版数チェック(週次, stub)
  ├── 通知: LINE / 週次メール (Secrets未設定)
  └── CI/CD: TSC + pytest + ESLint + Security Scan
```

### Gemini への問い
1. 上記の API 消費ポイントで、さらに削減できるものはあるか？
2. 「PDF 分類」をスクレイピング時ではなく、ユーザーが初めてその規制を閲覧した時に遅延実行する（Lazy Classification）方式は有効か？
3. テストカバレッジの優先順位: Python スクレイパー vs フロントエンドコンポーネント、どちらを先にテストすべきか？
4. 2,258行の publication_requirements.py は分割すべきか、それとも「単一ファイル=見通しの良さ」を優先すべきか？
